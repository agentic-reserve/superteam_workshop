# How to Design a Better Pitch Deck for Solana Projects

> Adapted from Kevin Hale's YCombinator presentation for Web3 and Solana ecosystem

## Introduction

Whether you're pitching to VCs, applying for grants (Solana Foundation, Superteam), or presenting at hackathons, your pitch deck needs to communicate clearly. This guide will help you create presentations that investors and judges actually remember.

## The Core Problem

After seeing 100+ Solana projects at a hackathon or demo day, investors will be lucky to remember 1-2 key points about your project. Your job is to make those points impossible to forget.

**The Three Rules:**

1. **Make it legible** - If they can't read it, they won't understand it
2. **Make it simple** - One slide = one idea
3. **Make it obvious** - Understandable at a glance

## Rule 1: Legibility

### Why It Matters

At Solana Breakpoint, Superteam events, or virtual pitch sessions, not everyone has perfect viewing conditions. Some judges are watching on mobile devices, others from the back of a conference hall.

### How to Make It Legible

- **Use large type** (minimum 32pt for body text, 48pt+ for headlines)
- **Bold text** for emphasis
- **Simple fonts** (Inter, SF Pro, Helvetica)
- **High contrast** (white text on dark background, or vice versa)
- **Text at the top** of slides (easier to read from distance)

### Bad Example

```
Small gray text at the bottom explaining your Solana program...
[Screenshot of complex Anchor code]
```

### Good Example

```
WE BUILT A SOLANA PROGRAM THAT
REDUCES TRANSACTION COSTS BY 90%
```

## Rule 2: Simplicity

### The One Idea Rule

Each slide should express exactly ONE idea. Don't try to explain your entire Solana architecture, tokenomics, and go-to-market strategy on a single slide.

### Ideal Slide Count

For a 2-3 minute pitch: **5-7 slides maximum**

1. Problem
2. Solution (what you built)
3. Traction/Demo
4. Market opportunity
5. Team
6. Ask (funding/grant amount)

### Bad Example

```
OUR SOLANA DEFI PROTOCOL

- Built on Solana for speed and low fees
- Uses Jupiter for swaps and Helius for RPC
- Implements compressed NFTs via Light Protocol
- Integrates with Phantom and Solflare wallets
- Mobile app coming Q2 2025
- Token launch planned for Q3
- Partnerships with 3 DAOs
```

### Good Example

```
DEFI PROTOCOL FOR SOLANA
THAT MAKES YIELD FARMING
AS EASY AS BUYING COFFEE
```

## Rule 3: Obviousness

### The Glance Test

Show your slide to someone unfamiliar with Solana. If they don't immediately understand your point, you lose.

### Why Speed Matters

- You only have 2-3 minutes
- Judges are easily distracted
- They're evaluating 50+ projects
- If they check their phone, your slide should still make sense when they look back up

### Making Ideas Explicit

Don't make judges guess what your metrics mean.

**Bad:**
```
[Graph showing upward trend]
```

**Good:**
```
10X GROWTH IN 30 DAYS
[Graph showing upward trend]
```

**Better:**
```
FROM 100 TO 10,000 USERS IN 30 DAYS
[Graph showing upward trend]
```

## Solana-Specific Tips

### Showing Technical Architecture

**Don't:**
- Show complex Anchor program diagrams
- Display raw Solana transaction logs
- Include full smart contract code

**Do:**
- Use simple flow diagrams (User → Program → Result)
- Show before/after comparisons
- Highlight the innovation in plain English

### Example: Explaining Your Solana Program

**Bad:**
```
[Screenshot of Anchor code with 50 lines]
```

**Good:**
```
OUR PROGRAM DOES 3 THINGS:

1. Accepts SOL deposits
2. Generates yield via Jupiter swaps
3. Returns profits to users

ALL IN ONE TRANSACTION
```

### Demonstrating Traction

**Bad:**
```
[Complex dashboard with 20 metrics]
```

**Good:**
```
$2M TOTAL VALUE LOCKED
IN 2 WEEKS SINCE LAUNCH
```

## The Screenshot Problem

Screenshots are almost always:
- Illegible (text too small)
- Complex (too many UI elements)
- Non-obvious (requires study to understand)

### Instead of Screenshots

Use **bulleted steps** or **simple descriptions**:

```
HOW IT WORKS:

1. Connect Phantom wallet
2. Deposit SOL
3. Earn 12% APY automatically
```

This is clearer than any screenshot of your dApp interface.

### Exception: Showing Complexity

The only time to show complexity is when you want to emphasize how hard the problem is that you're solving.

Example: If you're building a Solana aggregator that integrates 50+ protocols, showing all 50 logos makes the point that you're solving a real discovery problem.

## Common Mistakes in Solana Pitches

### 1. Too Technical

**Bad:** "We use Anchor's CPI to invoke Jupiter's swap instruction with a PDA-derived authority..."

**Good:** "We built a Solana program that automatically finds the best swap prices"

### 2. Buzzword Overload

**Bad:** "Leveraging Solana's parallel runtime with state compression and ZK proofs for a decentralized, trustless, permissionless..."

**Good:** "Fast, cheap NFT minting on Solana"

### 3. Comparing to Ethereum

Don't waste slides explaining why Solana is better than Ethereum. Judges already know. Focus on YOUR innovation.

### 4. Showing Wallet Addresses

Never show:
- Your program ID
- Wallet addresses
- Transaction signatures

Unless it's to prove a specific point (like "we processed 1M transactions").

## Solana Demo Day Checklist

Before you present:

- [ ] Every slide passes the "glance test"
- [ ] Text is readable from 20 feet away
- [ ] No more than 7 slides
- [ ] Each slide = one idea
- [ ] No screenshots (unless absolutely necessary)
- [ ] No complex diagrams
- [ ] No animations or transitions
- [ ] Metrics are explicit ("10X growth" not just a graph)
- [ ] Technical terms explained in plain English
- [ ] Clear ask at the end (funding amount, grant size, etc.)

## Tools for Creating Slides

- **Pitch** (pitch.com) - Modern, clean templates
- **Figma** - Full design control
- **Google Slides** - Simple and collaborative
- **Canva** - Easy templates

**Avoid:** PowerPoint (unless you're a design expert)

## Final Thoughts

Remember: Investors invest in teams, not slides. Your slides should make your ideas clearer, not distract from what you're saying.

**The goal isn't to impress with fancy slides. The goal is to be remembered.**

Make it:
- **Legible** - They can read it
- **Simple** - One idea per slide
- **Obvious** - Understandable at a glance

Do this, and you'll be better than 99% of Solana project pitches.
